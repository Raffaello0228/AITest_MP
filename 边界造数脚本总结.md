# Stage-Media 边界造数脚本总结

## 1. 背景

当前 MP 场景的核心问题是：用户反馈不同 `stage` 下的 `media` 占比波动较大，结果不够自然。  
为验证本期 PRD 的约束策略（尤其是 stage/media 相关约束与 KPI 达成规则），需要一套可复现的“边界用例”构造方式。

此前造数方式更偏“预设参数 + 启发式判断”，可解释性和稳定性一般。  
现在采用更直接的方法：**基于最终求解结果反推输入目标**，构造“刚好达标/边界达标”的 case。

---

## 2. 脚本功能

脚本：`tools/generate_stage_media_relaxation_case.py`

当前版本是精简版，只保留一个主流程：

1. 读取基准 brief；
2. 读取最终结果 `mp_result_*.json`；
3. 从指定维度（`corporation/category/ai`）提取明细；
4. 可选回填 `stage/marketingFunnel/media` 占比；
5. 反推 KPI 边界目标（按达成率阈值，或 must-achieve）；
6. 输出新的 brief，供再次执行与验收。

---

## 3. 核心逻辑

### 3.1 维度提取

- 从 `data.result.dimensionMultiCountryResult.*.<check_dimension>` 提取明细；
- 过滤 `TTL` 聚合行，避免重复统计；
- 支持按 `country_code` 过滤单国家。

### 3.2 占比回填（可选）

基于最终结果中的 `totalBudget` 聚合：

- `stage` 预算与比例
- `marketingFunnel` 预算与比例
- `media` 父层预算与比例
- `mediaChannel` 子层预算与比例

将这些值回写到 brief 中对应字段，使输入更贴近真实可行解。

### 3.3 KPI 边界反推

KPI 反推键：`Impression / Clicks / VideoViews`。

- **达成率模式**（默认）：给定 `kpi_target_rate`，反推一个“刚好达标”的 `target`；
- **must-achieve 模式**：直接设置 `target = actual`，并将 completion 设为必须达成。

同时会把 `kpiInfoBudgetConfig.rangeMatch` 以及 `mediaMarketingFunnelFormatBudgetConfig` 的灵活度同步为目标阈值，保持口径一致。

---

## 4. 输入输出

### 输入

- `--input-brief`：基准 brief（默认 `brief_stage_media_stablility.json`）
- `--result-file`：最终结果文件（必填）
- `--check-dimension`：`corporation` / `category` / `ai`
- `--kpi-target-rate`：边界达成率，默认 `95`
- `--kpi-must-achieve`：是否走 must-achieve 反推
- `--country-code`：可选，单国家过滤
- `--no-backfill-ratios`：可选，关闭占比回填

### 输出

- `--output-brief`：新生成 brief（默认输出到 `output/xiaomi/requests_stage_media_stability_from_brief/brief_case_boundary_from_result.json`）

---

## 5. 推荐用法

### 场景 A：AI 维度 + 90% 边界

```bash
python tools/generate_stage_media_relaxation_case.py \
  --result-file output/xiaomi/results/20260330_manual/mp_result_relaxation_trigger.json \
  --check-dimension ai \
  --kpi-target-rate 90
```

### 场景 B：单国家（SG）+ must-achieve 边界

```bash
python tools/generate_stage_media_relaxation_case.py \
  --result-file output/xiaomi/results/20260330_manual/mp_result_relaxation_trigger.json \
  --check-dimension ai \
  --country-code SG \
  --kpi-must-achieve
```

---

## 6. 适用边界与注意事项

- 该脚本用于“构造边界输入”，不是直接判定算法正确性；
- 最终是否通过，仍以执行后的 `achievement_check` 结果为准；
- 若某维度下 KPI 实际值为 0，反推目标也会是 0，这是数据现象，不是脚本异常；
- 若需要精确比对“刚过线/刚不过线”，建议再配合执行脚本批量跑两组相邻阈值。

---

## 7. 对话上下文压缩记录

### 7.1 需求演进（压缩）

- 初始目标：围绕 PRD（stage 内 media 波动过大）产出测试用例；
- 中期目标：解释造数脚本为何能触发场景，并纠正“必然失败”的表述；
- 最终目标：从“启发式造数”切换为“基于最终结果反推边界 case”，并支持 `AI` 维度执行。

### 7.2 关键结论（压缩）

- “严格容差一定失败”不严谨，应改为“高概率失败”，最终以实际执行结果为准；
- 更稳的方法是基于 `mp_result` 回填占比并反推 KPI 边界目标；
- 维度选择会显著影响反推结果，`corporation` 和 `ai` 的可用行与 KPI 数值可能不同。

### 7.3 脚本改造轨迹（压缩）

- 删除了旧版多分支逻辑（固定配比、复杂松弛触发链路）；
- 重写为精简版主链路：`读取结果 -> 维度提取 -> 占比回填 -> KPI边界反推 -> 输出brief`；
- 新增参数 `--check-dimension`，支持 `corporation/category/ai`；
- 修复边界搜索性能问题：由大范围遍历改为“小窗口近似搜索”，避免在大 KPI 下运行卡住。

### 7.4 已验证执行（压缩）

- 已按 `AI` 维度 + `kpi_target_rate=90` 成功运行；
- 生成文件：`output/xiaomi/requests_stage_media_stability_from_brief/brief_case_boundary_from_result.json`；
- 回填与反推均生效（以脚本运行输出 summary 为准）。
